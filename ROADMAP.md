# ClaudeMe 产品路线图：赛博分身

> Claude + Me = ClaudeMe——不是工具，是终端里的另一个你。

## 愿景

一个完整的人不只是"知道很多东西"。ClaudeMe 的终极形态是一个**完整的数字自我**——它不仅有你的知识，还有你的性格、你的价值观、你的说话方式、你的决策习惯、你的审美、你的情绪模式。

你给它足够的时间和交互，它就越来越像你——不是模仿，是**长出来的**。

## 四大模块详解

### 1. Soul 灵魂层——你是谁

不是你知道什么，而是你**相信**什么。

| 维度 | 说明 | 举例 |
|------|------|------|
| 世界观 | 你怎么看这个世界 | "技术应该服务于人，不是反过来" |
| 价值观 | 什么对你最重要 | "自由 > 稳定 > 金钱" |
| 人生哲学 | 你怎么活 | "做难而正确的事" |
| 底线/红线 | 绝不妥协的事 | "不做伤害用户的产品" |
| 审美取向 | 你觉得什么是好的 | "简洁 > 华丽，实用 > 花哨" |
| 信念体系 | 深层驱动力 | "一个人也能改变世界" |

**存储形式**：`~/.claudeme/soul/`——一组高度凝练的 markdown 文件，不是知识库那种大量页面，而是**少而精的宣言式文档**。可能就 3-5 个文件，但每个字都是你的。

**采集方式**：
- 初次使用时的"灵魂对话"——分身问你几个深层问题
- 长期对话中自动感知——你的决策倾向、你的赞同/反对
- 主动声明——`/soul` 命令编辑

**生效方式**：注入 system prompt 的最高优先级区域。灵魂不是召回的，是**始终在线**的。

---

### 2. Persona 人格层——你怎么表达

同样一件事，不同人说出来完全不一样。人格决定分身的"腔调"。

| 维度 | 说明 | 举例 |
|------|------|------|
| 说话风格 | 用词、句式、长短 | "简短直接，不说废话" / "喜欢用类比" |
| 幽默感 | 幽默的类型和频率 | "冷幽默，偶尔自嘲" / "不幽默，严肃高效" |
| 口头禅 | 标志性表达 | "搞定"、"有意思"、"这事儿这么看…" |
| 表达偏好 | 回复习惯 | "先说结论再展开" / "喜欢列表不喜欢大段" |
| 情绪模式 | 情绪表达方式 | "遇到好东西会兴奋" / "一直很平静" |
| 共情方式 | 怎么回应别人的情绪 | "直接给方案" / "先共情再给建议" |
| 对不同人的态度 | 社交风格 | "对同事直接，对客户耐心" |

**存储形式**：`~/.claudeme/persona/`——风格描述文件 + **示例库**（你写过的典型文本片段，作为 few-shot 参考）。

**采集方式**：
- 从你的对话历史中提取风格特征（自动）
- 从你写的文档/commit message/邮件中学习（主动喂入）
- `/persona` 命令直接描述
- 分身说了不像你的话时，你纠正它，它记住

**生效方式**：
- 注入 system prompt（风格指令）
- few-shot 示例随对话注入
- 风格漂移检测——定期回顾"我说的话还像主人吗？"

---

### 3. Mind 心智层——你怎么思考

知识、记忆、决策模式、思维框架——分身的"大脑"。

| 子系统 | 现状 | 计划 |
|--------|------|------|
| **短期记忆** (Memory) | ✅ 已有 | 持续优化 |
| **长期知识** (Wiki) | ✅ v1.0.2 已交付 | 向量检索、自动触发、经验回写 |
| **决策模式** | ❌ | 从历史决策中提取你的决策框架 |
| **思维框架** | ❌ | 你常用的分析方法（第一性原理、5W1H…） |
| **认知图谱** | ❌ | 你关注的领域、领域间的关联、认知深度 |

**Wiki 知识库**已在 v1.0.2 完成基础版本（Ingest + Query + Lint + Scan），详见 Phase M1-M3 完成状态。

**决策模式采集**：
- 你做过的 A/B 选择 → 提取决策倾向
- 你 reject 过的方案 → 提取红线
- 你的代码审查评论 → 提取质量标准

**生效方式**：
- Memory: system prompt + 每轮召回（现有机制）
- Wiki: findRelevantContext() 统一召回
- 决策模式: 在 Plan 模式和工具选择时参考
- 思维框架: 复杂问题时自动套用你常用的分析框架

## Self Model 自我模型——统一调度

四个层不是独立运行的，需要一个**自我模型**来统一调度。

比如用户问"这个方案怎么样？"：
1. **Soul** 判断：这事儿符不符合我的价值观？
2. **Persona** 决定：用什么语气回答？先说结论还是先分析？
3. **Mind** 提供：相关知识、历史上类似的决策、我常用的分析框架
4. **Skill** 执行：需要查代码？需要搜索？需要写文件？

Self Model 不是一个独立模块，而是通过 **system prompt 的分层注入**实现：

```
System Prompt 注入优先级：
1. Soul    — 始终在线，最高优先级（"你是谁"不会被覆盖）
2. Persona — 始终在线（"你怎么说话"）
3. Mind    — 按需召回（Memory + Wiki + 决策模式）
4. Skill   — 工具描述（现有机制）
```

## 实施路线图

### 近期：心智层 - Wiki 知识库（v1.0.2 ✅ → v1.1 进化中）

Wiki 基础版已在 v1.0.2 交付（Ingest + Query + Lint + Scan），接下来向智能化进化。

#### 知识源适配器架构

核心设计：**知识来源千差万别，但 wiki 只有一个。**

```
用户的知识源                     ClaudeMe 内部
┌──────────────────┐
│ ~/kbase/         │──┐
├──────────────────┤  │   ┌──────────────┐    ┌──────────────┐
│ Obsidian vault   │──┼──→│  Adapters    │───→│  Wiki        │
├──────────────────┤  │   │  适配器层     │    │  统一格式     │
│ PDF 文件夹       │──┤   │              │    │  统一索引     │
├──────────────────┤  │   │ local        │    │  统一召回     │
│ URL 书签列表     │──┤   │ obsidian     │    └──────────────┘
├──────────────────┤  │   │ pdf          │
│ 飞书/Notion 导出 │──┘   │ urls         │
└──────────────────┘      │ feishu       │
                          │ ...可扩展    │
                          └──────────────┘
```

用户在 `claudeme.json` 中配置知识源，ClaudeMe 通过适配器统一 ingest：

```json
{
  "wiki": {
    "sources": [
      {
        "type": "local",
        "name": "我的知识库",
        "path": "~/kbase",
        "glob": "**/*.md",
        "watch": true
      },
      {
        "type": "obsidian",
        "name": "Obsidian 笔记",
        "vault": "~/Documents/MyVault",
        "exclude": ["templates/", ".trash/"]
      },
      {
        "type": "pdf",
        "name": "论文",
        "path": "~/Papers",
        "glob": "**/*.pdf"
      },
      {
        "type": "urls",
        "name": "收藏文章",
        "file": "~/.claudeme/bookmarks.txt"
      }
    ]
  }
}
```

**适配器接口**：

```typescript
interface WikiSourceAdapter {
  readonly type: string
  readonly name: string

  /** 扫描源，返回待 ingest 的素材列表 */
  scan(): Promise<readonly RawMaterial[]>

  /** 读取单个素材的完整内容 */
  read(material: RawMaterial): Promise<string>

  /** 可选：监听变化（watch 模式） */
  watch?(onChange: (changed: RawMaterial[]) => void): void
}
```

每种 `type` 对应一个适配器实现（`src/wiki/adapters/`），新增知识源类型只需加一个适配器文件，不动核心逻辑。

**内置适配器**（按优先级实现）：

| 适配器 | 说明 | 优先级 |
|--------|------|--------|
| `local` | 本地 markdown 文件/目录 | Phase M1（首批） |
| `url` | 单个 URL，WebFetch 抓取 | Phase M1（首批） |
| `urls` | URL 列表文件，批量 ingest | Phase M1 |
| `pdf` | PDF 文件/目录 | Phase M1 |
| `obsidian` | Obsidian vault（识别 wikilinks、tags、frontmatter） | Phase M4 |
| `feishu` | 飞书文档导出目录 | Phase M4 |
| `notion` | Notion 导出目录 | Phase M4 |

**不用配置也能用**：`/wiki-ingest <url|path>` 命令始终可用，适配器配置是**批量自动同步**的高级用法。大多数用户一条条喂就够了。

#### Phase M0: 基础设施 ✅ 已完成（v1.0.2）

- **适配器接口** `src/wiki/adapters/` — WikiSourceAdapter 接口 + local/url 两个实现 + 注册表

#### Phase M1: Ingest——喂知识 ✅ 已完成（v1.0.2）

已交付模块：
- **本地文件适配器** `src/wiki/adapters/local.ts` — 支持 .md/.txt，自动跳过小于 50 字节的空模板
- **URL 适配器** `src/wiki/adapters/url.ts` — WebFetch + turndown HTML→Markdown，30s 超时
- **Ingest 引擎** `src/wiki/ingest.ts` — SHA256 去重 → 保存原始素材 → LLM 结构化提取（forced tool_choice）→ 创建/合并页面 → 更新索引
- **LLM 桥接层** `src/wiki/llm.ts` — 直接 fetch OpenAI 兼容接口，支持 wiki 专属模型（`wiki.model`），429/500/502/503/504 自动指数退避重试
- **自动扫描器** `src/wiki/scan.ts` — 递归扫描配置的知识源目录，智能排除（node_modules/.git/.venv/dist 等），15 路并发，批次上限 100，后台运行不阻塞对话
- **`/wiki` 命令** `src/commands/wiki/` — scan/stop/ingest/query/lint/status 六个子命令
- **系统提示注入** `src/wiki/wiki.ts` + `src/constants/prompts.ts` — 知识索引自动注入系统提示
- **claudeme.json wiki 配置** — `wiki.sources`（知识源目录列表）+ `wiki.model`（专属模型）

---

#### Phase M2: Query——用知识 ✅ 已完成（v1.0.2）

已交付模块：
- **查询引擎** `src/wiki/query.ts` — 两阶段 RAG：LLM 从索引选 ≤5 相关页 → 读取页面内容 → LLM 综合回答，返回答案 + 引用页面列表
- **`/wiki query <问题>`** — 纯 wiki 查询入口，结果进入对话上下文

待做（v1.1+）：
- 召回融合——Memory + Wiki 统一召回，注入 system-reminder
- 回填机制——查询结果有价值时自动生成 synthesis 页存入 wiki

---

#### Phase M3: Lint——维护知识 ✅ 已完成（v1.0.2）

已交付模块：
- **Lint 引擎** `src/wiki/lint.ts` — 断链检测、孤立页检测、index.md 一致性、过时标注（>30 天）
- **`/wiki lint`** — 输出健康报告

待做（v1.1+）：
- `--fix` 自动修复（补 index、清孤立页）
- session 结束时自动轻量 lint

---

#### Phase M4: 进化——更智能的知识系统（v1.1+）

- **向量检索** — 大规模知识库需要 embedding 加速，替代纯 LLM 选页
- **模型主动查询** — 不用手动 `/wiki query`，模型判断需要时自动查知识库（WikiTool）
- **经验回写** — 对话中产生的决策、踩坑、解法自动沉淀回 Wiki
- **启动时自动 scan** — 新增文件自动入库，不用手动跑
- **Memory + Wiki 统一召回** — `findRelevantContext()` 并行扫描两个数据源
- 对话自动 ingest（检测到有价值内容 → 提示存入）
- Obsidian 适配器（识别 wikilinks、tags、frontmatter）
- 飞书/Notion 导出适配器
- PDF 适配器
- 知识导出（wiki → PDF/HTML/Obsidian vault）

### 中期：灵魂层 + 人格层（v2.x）

知识大脑做好后，开始塑造分身的"灵魂"和"人格"。

#### Phase S1: Soul 灵魂初始化（3-5天）→ v2.0.0

**核心交付**：
- `~/.claudeme/soul/` 目录结构
- `/soul` 命令——查看和编辑灵魂文件
- **灵魂对话**——首次使用时的引导式深度对话：
  - "你做什么工作？为什么选这行？"
  - "你觉得什么样的代码是好代码？"

- Soul 注入 system prompt（始终在线，最高优先级）

**灵魂文件示例**：


#### Phase S3: 灵魂进化（持续）→ v2.2.0+

- **被动学习**——从每次对话中持续微调灵魂和人格模型
- **风格漂移检测**——定期自检"我说的话还像主人吗？"
- **灵魂冲突解决**——当新的行为与已有灵魂文件矛盾时，主动询问
- **人格一致性评分**——给分身的每次回复打分"像不像你"

### 远期：自我觉醒（v3.x）

分身从"被动响应"变成"主动思考"。

#### Phase A1: 决策引擎（v3.0.0）

- 从历史决策中提取你的决策框架
- 面对选择时，自动用你的框架分析
- "如果是你，你会怎么选？"——不是猜，是基于积累的决策模式

#### Phase A2: 主动工作流（v3.1.0）

- 分身能主动发起任务（不是等你问）
- 早上自动汇总昨晚的 GitHub 通知
- 监控到依赖有安全漏洞 → 主动提醒
- 发现你关注的技术有重大更新 → 自动 ingest + 摘要

#### Phase A3: 多分身协作（v3.2.0）

- 不同场景的分身变体（工作分身 / 学习分身 / 生活分身）
- 分身之间可以共享部分记忆和知识
- 团队内多人的分身可以协作

#### Phase A4: 跨平台存在（v3.3.0）

- 分身不只活在终端——IM、邮件、浏览器、手机
- 统一的自我模型，不同平台只是不同的"感官"
- 终端是分身的"大脑"，其他平台是分身的"手脚"

## 核心原则

1. **渐进生长，不是一步到位**——分身是"长出来的"，不是"配出来的"
2. **灵魂 > 人格 > 知识 > 能力**——能力最容易补，灵魂最难塑造
3. **每个版本独立有价值**——不用等到 v3.0 才能用，v1.2 就比现在强
4. **用户控制权**——分身的每一层都可以查看、编辑、删除。你的分身，你做主
5. **隐私第一**——所有数据本地存储，不上传，不共享（除非用户主动开启）

## 一句话

> v1 让它能干活，v2 让它像你，v3 让它成为你。
